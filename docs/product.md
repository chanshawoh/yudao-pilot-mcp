# Yudao Pilot 产品说明

## 产品概述

Yudao Pilot 是一个面向 yudao 生态的工作区感知型 MCP 服务。它的作用不是替代 AI 应用，而是为 AI 应用补上“理解 yudao 项目结构、读取本地数据库配置、决定代码应该落到哪里”的关键能力。

第一版强调实用和可控，不追求全自动自治，而是先成为 AI 应用和真实 yudao 项目之间的稳定桥梁。

## 要解决的问题

通用型 AI 编码工具在 yudao 项目里通常会卡在三个地方：

- 不知道当前目录是不是一个合法的 yudao 工作区
- 不知道当前到底有哪些后端和前端项目类型
- 不知道一张表对应生成的代码该落到哪个模块、哪个项目

这会带来很直接的交付风险：

- 代码写错模块
- 前端目标猜错
- 数据库配置推断错误
- AI 应用反复追问用户目录结构和项目信息

## 产品目标

通过一个确定性的 MCP 层，让 AI 应用能够：

- 判断当前目录是否已完成 yudao 工作区初始化
- 安全读取项目结构和数据库配置
- 推导某张业务表对应的模块、业务名、实体名
- 把生成结果落到正确的后端和前端项目中

## 目标用户

- 使用 yudao 的 Java / Vue 开发者
- 追求快速交付后台业务系统的小团队
- 在 Cursor、VS Code、Trae 等编辑器中使用 AI 开发的团队
- 希望让代码生成流程更稳定，但暂时不想自建内部平台的团队

## 产品原则

- 配置优先：工作区配置是唯一事实来源
- 严格校验：不支持或不明确的情况尽早失败
- 人机协同：AI 应用负责和用户对话，Yudao Pilot 负责结构和路由
- 范围收敛：先把最核心流程做稳，再考虑自动化扩展

## 第一版产品定义

第一版支持：

- 当前目录工作区识别
- `./.yudao-pilot/config.yaml` 的初始化与解析
- 1 个后端项目和多个不重复前端类型的校验
- 从配置或后端本地环境中解析数据库连接
- 基于模块、表前缀、精确表规则的手动路由
- 面向 AI 应用的生成规划与文件落盘能力

第一版暂不支持：

- 数据库副本机制
- 表结构合并与数据合并
- 多环境发布流水线
- 报错驱动的自动修复重试闭环

## 工作区模型

每个工作目录都可以视为一个 yudao-pilot 工作区。

工作区由以下元素组成：

- 一个根目录
- 一个 `.yudao-pilot/config.yaml` 配置文件
- 严格唯一的后端项目
- 0 个或多个前端项目，且前端类型不能重复

这样设计的好处是：AI 应用不用猜，只需要读取配置和校验结果，就能明确当前工作区到底支持什么。

## 核心使用流程

1. AI 应用进入当前工作目录。
2. 调用 MCP 服务加载工作区配置。
3. 如果配置不存在，Yudao Pilot 返回初始化模板。
4. AI 应用引导用户完成首次配置。
5. Yudao Pilot 校验后端和前端项目。
6. AI 应用询问或识别目标业务表。
7. Yudao Pilot 解析业务规则并返回生成规划。
8. AI 应用生成代码内容。
9. Yudao Pilot 把生成结果写入正确项目位置。

## 商业策略

建议采用开放核心模式。

社区版：

- MCP stdio 接入
- 工作区配置初始化
- 项目识别与校验
- 数据库配置解析
- 标准代码生成路由与落盘

未来付费方向：

- 更丰富的行业模板和生成模板包
- 团队级治理能力，例如生成结果审查和规范控制
- 多环境工作流编排
- 特定业务场景的模板增强

## 为什么适合开放核心模式

- 基础能力只有先被 yudao 开发者广泛使用，产品才有价值
- 配置驱动的方式可以降低接入成本，利于社区传播
- 付费能力可以聚焦团队协作和行业增强，而不是限制协议层基础能力

## 命名与合规

产品继续使用独立名称 Yudao Pilot，并明确描述为面向 yudao 生态的社区增强工具。

这样既能降低命名混淆，也能保持与 yudao 开源生态的兼容性和亲和力。

## 产品风险

- 不同 yudao 版本之间的结构差异，可能会让规则识别需要更多适配
- 各团队的本地配置方式不完全一致，数据库解析需要做足兼容
- 最终代码质量仍然依赖外部 AI 应用的生成能力
- 如果配置约束不够严格，用户容易期待“全自动推断”，超出第一版边界

## 推荐定位语

Yudao Pilot 是帮助 AI 应用理解 yudao 工作区、解析本地数据库上下文并将生成代码准确落到正确项目中的 MCP 桥梁。
