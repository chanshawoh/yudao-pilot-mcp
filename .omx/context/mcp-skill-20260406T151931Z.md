# Context Snapshot: mcp-skill

- Task statement: 把当前 `yudao-pilot-mcp` 的 MCP 能力封装成一个可复用的 Skill。
- Desired outcome: 产出一个面向 AI/Codex 使用的 Skill，让调用者能按正确顺序使用当前 MCP 能力，而不是每次临时理解仓库和工具。
- Stated solution: 基于当前项目已有 MCP 工具，整理并封装为一个 Skill。
- Probable intent hypothesis: 用户希望把仓库里已经沉淀的 Yudao Pilot MCP 能力产品化为标准技能入口，降低后续使用门槛，并把正确使用方式固化下来。
- Known facts/evidence:
  - 当前 MCP 入口在 `src/yudao_pilot/server.py`，服务名为 `Yudao Pilot`。
  - 已注册工具包括：`load_workspace_config`、`init_workspace_config`、`inspect_project_path`、`compare_codegen_reference_projects`、`validate_workspace_projects`、`resolve_database_config`、`infer_codegen_plan`、`inspect_codegen_context`、`inspect_table_schema`、`generate_codegen_scaffold`、`write_generated_files`、`write_mysql_migration`、`generate_codegen_sql`。
  - 项目已有 `.ai/ai-integration.md`，其中定义了推荐调用顺序和 SQL/菜单图标等接入协议。
  - 项目已有 `.ai/bootstrap.md` 与 `.ai/dev-standards.md`，约束了工作区配置、项目识别、代码生成、写入规则和联调约定。
- Constraints:
  - Skill 需要忠实反映当前 MCP 的真实能力边界，不能虚构不存在的工具或流程。
  - Skill 应适配 Codex 技能格式，并尽量减少后续使用者的上下文成本。
  - 当前用户请求是“封装成一个技能”，还未明确是偏“操作手册型”还是“自动执行型”。
- Unknowns/open questions:
  - 这个 Skill 的目标用户是谁：AI 助手自身、仓库维护者，还是业务侧使用者？
  - Skill 的默认职责边界应该停在“指导调用 MCP”，还是允许“直接代表用户执行 codegen + 写入 + SQL”？
  - Skill 产物希望放在当前仓库，还是安装到全局 `$CODEX_HOME/skills` 下复用？
- Decision-boundary unknowns:
  - 哪些参数允许 Skill 自行推断，哪些必须在执行前向用户确认？
  - 当配置缺失、数据库不可连、表不存在时，Skill 应该停下提问，还是继续走只读分析路径？
- Likely codebase touchpoints:
  - `src/yudao_pilot/server.py`
  - `.ai/ai-integration.md`
  - `.ai/bootstrap.md`
  - `.ai/dev-standards.md`
  - `README.md`
