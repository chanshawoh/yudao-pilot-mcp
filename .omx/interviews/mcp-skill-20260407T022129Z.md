## Outcome

用户要的不是“指导型” Skill，而是“执行型” Skill：

- 开发者一旦安装该 Skill，生成型请求默认优先走这个 Skill
- Skill 以当前 `yudao-pilot-mcp` 的真实 MCP 能力为执行入口
- 当前阶段不在 Skill 内加入 MCP 安装说明

## Key Clarifications

### Intent

- Skill 要把当前 MCP 的能力封装成默认执行通道，而不是让 AI 绕过 MCP 自己拼代码

### Scope

- 第一版范围选择 `C`
- 即：当前 MCP 的全部能力都纳入 Skill，而不只封装主链路

### Non-goals

- 不负责联调启动后端/前端
- 不负责修改 `.ai/` 协议文件
- 不在没有 `write_files` 意图时真实写入
- 不替代人工确认高风险数据库写操作
- 不在当前阶段加入 MCP 安装说明

### Decision Boundaries

- 配置缺失时，不必确认，自动生成默认可用配置
- `module_name / business_name / entity_name` 可以自动推导，不必确认
- 数据库无法连接时，直接停止运行，要求用户配置数据库连接
- 数据库表不存在，且找不到迁移文件时，直接停止运行，要求用户先创建表

## Pressure Pass

回访的核心假设：

- “安装了这个 Skill 就必定要用这个能力”的前提，是它必须比自由发挥更稳定、更可检验

用户给出的最小成功证据：

- 开发者只给 `table_name`，Skill 就能自动走完 `load_config/init -> validate -> resolve_db -> inspect/generate` 主链路
- 遇到数据库不可连时，Skill 不会误写入
- 生成型请求默认优先走 Skill，而不是跳过 MCP 自己拼代码
