## Intent

Keep code generation aligned with the original Java codegen naming rules, while changing the default SQL execution control semantics so menu/dict SQL is still generated by default but database execution is gated by a workspace configuration switch with highest priority.

## Desired Outcome

The generator should:
- continue generating menu/dict migration SQL by default
- stop relying on the tool parameter `apply_menu_to_database`
- introduce a workspace-level top-priority config switch controlling whether menu/dict SQL may be executed against the database
- preserve original Java codegen naming-rule precedence
- update tests and README to match

## In Scope

- Code logic changes for menu/dict SQL apply control
- Removal of `apply_menu_to_database` logic from the tool flow
- New workspace config switch for DB-apply permission
- Tests covering the new default/control semantics
- README/config documentation updates
- Naming-rule handling only insofar as needed to preserve “original Java codegen rules first”

## Out of Scope / Non-goals

- Do not stop generating menu/dict migration SQL by default
- Do not introduce a new custom naming convention that overrides original Java codegen behavior
- Do not change frontend CRUD/template alignment behavior from the previous branch
- Do not expand into unrelated backend/controller/template refactors beyond what is required by the SQL-control change

## Decision Boundaries

- Original Java codegen naming rules remain authoritative
- Workspace config has higher priority than any runtime/tool request for DB apply
- `apply_menu_to_database` tool parameter logic should be removed rather than retained as an override path
- One total switch controls both menu and dict DB execution permission

## Constraints

- Preserve default generation of menu/dict migration SQL
- Keep behavior explicit and documented
- Update README and tests in the same branch as the logic change
- Avoid unrelated generator changes

## Testable Acceptance Criteria

1. Menu/dict migration SQL is still generated by default.
2. Database execution of menu/dict SQL is controlled by a new workspace configuration switch.
3. When the config switch disables DB apply, no runtime/tool path can still write menu/dict data to DB.
4. `apply_menu_to_database` no longer drives behavior in the tool flow.
5. README and config examples document the new behavior correctly.
6. Existing naming behavior remains aligned to original Java codegen precedence.
7. Regression tests cover the new config-precedence behavior.

## Assumptions Exposed + Resolutions

- Assumption: restoring naming-rule changes means restoring earlier custom naming ideas.
  - Resolution: rejected; naming still follows original Java codegen precedence.
- Assumption: explicit tool parameters should still be able to force database apply.
  - Resolution: rejected; config total switch has highest priority and tool-parameter logic should be removed.
